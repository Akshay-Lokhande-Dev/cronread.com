SQL Logical Execution Order — Complete Reference

The most important concept for understanding complex SQL queries is logical execution order. The database engine doesn't process SQL in the order you write it — it follows a fixed sequence.

StepClauseWhat happensCommon mistake
1FROM / JOINTables are identified and joined togetherAssuming ORDER BY applies before JOIN
2WHERERows filtered before any groupingUsing aliases defined in SELECT (not available yet)
3GROUP BYRemaining rows grouped into bucketsSelecting non-aggregated, non-grouped columns
4HAVINGGroups filtered using aggregate conditionsUsing HAVING instead of WHERE for row-level filters
5SELECTColumns computed and aliases definedReferencing a SELECT alias in WHERE (too early)
6DISTINCTDuplicate rows removedExpecting DISTINCT before GROUP BY
7ORDER BYResult set sortedSorting before LIMIT — always applies together
8LIMIT / OFFSETFinal row count restrictedUsing LIMIT without ORDER BY (non-deterministic)

Cloud Data Warehouse Cost Patterns

On pay-per-scan platforms like AWS Athena, Amazon Redshift Spectrum, and Snowflake, poorly written SQL directly increases cost. Key patterns to avoid:

Anti-PatternPlatform RiskFix
SELECT * on large tablesAthena, Redshift, SnowflakeList only needed columns — reduces bytes scanned
No partition filter in WHEREAthena, Redshift SpectrumAlways filter on partition key (e.g. event_date, region)
CROSS JOIN on large tablesAll platformsReplace with INNER JOIN or add join condition
Functions on indexed/partition columnsAll platformsAvoid WHERE YEAR(created_at) = 2024 — use range instead
Scanning non-columnar formatAthenaConvert to Parquet/ORC for 10x cost reduction
No LIMIT in dev/test queriesSnowflake, BigQueryAlways add LIMIT when exploring data

Frequently Asked Questions

SQL execution, cloud costs, and schema analysis explained.

What is SQL logical execution order?
SQL logical execution order is FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT. This is the sequence the database engine processes clauses — not the order you write them. Understanding this explains why you can't use a SELECT alias in a WHERE clause, or why HAVING is needed for aggregate filters.
Why does SELECT * cost more on AWS Athena?
Athena charges per terabyte of data scanned. If your data is in columnar format (Parquet or ORC), specifying only the columns you need means Athena reads only those column files — skipping the rest. SELECT * forces Athena to read every column. On wide tables with 50+ columns, this can be 10-20x more expensive than a targeted column list.
What is a partition filter and why does it matter?
On Athena and Redshift Spectrum, large tables are often partitioned by date, region, or another column. Without a WHERE clause filtering on the partition key, the query scans the entire table (all partitions). Adding a partition filter like WHERE event_date = '2024-01-15' limits the scan to only that partition — dramatically reducing cost and query time.
What is schema-aware SQL analysis?
Schema-aware analysis cross-references your SQL query against your CREATE TABLE definitions to find issues like missing indexes on JOIN columns, data type mismatches (e.g. BIGINT joined to INT), missing foreign keys, columns referenced in the query that don't exist in the schema, and column size concerns.
What does an ASCII SQL execution tree show?
The ASCII visual flow shows the order of data operations the database performs — which tables are read, how filters reduce row counts, how joins combine data, where aggregation happens, and what is finally returned. It mirrors a simplified query execution plan and helps you identify where performance bottlenecks are most likely to occur.
What is the difference between WHERE and HAVING?
WHERE filters individual rows before grouping — it runs at step 2 in execution order. HAVING filters groups after GROUP BY — it runs at step 4. A common performance mistake is using HAVING for row-level filters that could be WHERE, forcing the database to group all rows before discarding many of them.
How do I check if a JOIN will cause a data type mismatch?
Compare the column definitions in your CREATE TABLE statements for both sides of the JOIN condition. If users.id is BIGINT and orders.user_id is INT, the database must implicitly cast one type to the other on every row comparison — making index usage impossible on some engines. Always match data types exactly on JOIN columns.
