How to Test a Cron Expression

CronRead's cron tester works in three steps:

Cron Formats: 5-Field, 6-Field, and 7-Field

Different platforms use different cron formats. A good cron tester needs to know which format you're using:

FormatFieldsUsed byExample
Standard (POSIX) 5 Linux crontab, Kubernetes, GitHub Actions 0 2 * * *
AWS EventBridge 6 AWS EventBridge Scheduler, CloudWatch Events 0 2 * * ? *
Quartz (Java) 6–7 Spring Boot, Quartz Scheduler, many Java apps 0 0 2 * * ?
Jenkins 5 (with H) Jenkins pipeline triggers H 2 * * *
⚠ AWS EventBridge difference: AWS requires either day-of-month OR day-of-week to be set — not both. Use ? for the one you're not setting. A standard cron tester will reject valid AWS expressions if it doesn't understand this rule.

Debugging Common Cron Errors

Expression is valid but fires at the wrong time

This is almost always a timezone issue. The tester shows times in your local timezone by default, but your server or scheduler may run in UTC. Use the timezone selector in CronRead to switch to UTC and verify the actual fire time.

Day-of-month and day-of-week conflict

In standard cron, if you set both day-of-month and day-of-week (i.e. neither is *), cron uses OR logic — the job fires if either condition matches. This surprises many users. Example: 0 0 1 * 1 runs on the 1st of every month AND every Monday, not only on Mondays that are the 1st.

Expression never fires

Check for impossible dates like February 30th (0 0 30 2 *) — this expression will never fire. A next-run-time calculator immediately reveals this because it will show no upcoming runs.

Job runs unexpectedly often

A common mistake: * * * * * runs every single minute. If you meant "once per hour", you need 0 * * * * (minute field must be 0, not *).

Testing Cron Expressions by Platform

Different platforms implement slightly different cron dialects. Here is what to watch for when testing expressions for specific schedulers:

Linux crontab (5-field)

Standard 5-field format: minute hour day month weekday. Day-of-week uses 0–7 (0 and 7 are both Sunday). The system daemon reads /etc/crontab and user crontabs from /var/spool/cron/crontabs/. Verify your expression with crontab -l after editing. The timezone used is the system timezone — not UTC unless CRON_TZ is set in the crontab file.

AWS EventBridge Scheduler (6-field)

AWS uses a 6-field format with an extra Year field at the end: minute hour day month weekday year. The ? wildcard is required in either the day-of-month or day-of-week field (not both *). All times are interpreted as UTC only. Use the AWS EventBridge Cron Guide for platform-specific examples.

GitHub Actions (5-field, UTC)

GitHub Actions schedule triggers use standard 5-field cron and always run in UTC. Scheduled workflows may run up to 15 minutes late under high load. The minimum interval is every 5 minutes — GitHub will not guarantee runs more frequent than that. Test your expression in UTC to see the actual trigger window.

Kubernetes CronJob (5-field)

Kubernetes CronJobs support standard 5-field cron. Since Kubernetes 1.25, you can specify a timeZone field in the spec (e.g. timeZone: "America/New_York"); earlier versions always use UTC. The startingDeadlineSeconds field controls how late a missed run can start — worth testing with your expression to ensure recovery after cluster downtime.

Frequently Asked Questions

How do I test a cron expression for free?
Use CronRead — paste your expression in the tool at the top of this page. It shows next run times, plain-English description, and field validation instantly, for free, with no sign-up.
Can I test AWS EventBridge cron expressions?
Yes. AWS EventBridge uses 6-field cron with different rules for ? and L. CronRead supports AWS mode — see the AWS EventBridge guide for the differences.
What is the difference between a cron tester and a cron expression generator?
A tester validates an expression you already have and shows next run times. A generator helps you build an expression from a human-readable description (e.g. "every weekday at 9 AM"). CronRead does both.
Does */5 in the minute field mean every 5 minutes starting from midnight?
Yes — */5 in the minute field runs at minutes 0, 5, 10, 15, 20, 25, 30, 35, 40, 45, 50, 55. It always starts from 0 (the lowest value in the field's range).
Can cron run more than once a minute?
No — the finest granularity in standard cron is 1 minute. For sub-minute scheduling, use systemd timers, a task queue, or a custom scheduling loop inside your script.
